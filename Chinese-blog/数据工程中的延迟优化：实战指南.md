## 1. 什么是数据 Pipeline 中的延迟？

**延迟（Latency）** 是指一个事件在现实世界中发生，到该事件的结果出现在下游系统（看板、ML 模型、告警系统、API 接口）之间所经历的时间。

在数据工程语境下，延迟可以细分为三种类型：

| 术语 | 定义 | 示例 |
|---|---|---|
| **摄入延迟（Ingestion Latency）** | 数据从生成到落入存储层的时间 | 09:00:00 写入 Kafka 的消息，09:00:03 才落到 S3 |
| **处理延迟（Processing Latency）** | 计算层完成数据转换所花费的时间 | Spark 任务运行了 40 分钟 |
| **端到端延迟（End-to-End Latency）** | 从原始事件到最终可消费输出的总时间 | 09:00 发生的交易，09:45 才出现在仪表盘 |

同样重要的是，明确延迟**不是什么**：

- 延迟不等于**吞吐量**（单位时间内处理的数据量）。高吞吐量的系统依然可能有高延迟。
- 延迟问题不总是由"代码写得差"造成的，很多时候根源在于**上游的架构决策**。
- **"慢"只是症状，不是原因。** 根本原因可能在计算层、存储层、网络层或调度层——它们需要完全不同的解决方式。

## 2. 延迟 vs. 吞吐量：核心权衡

在调优任何参数之前，必须先理解这对根本性矛盾：

```
优化低延迟 ←────────────────────────→ 优化高吞吐量
（立即处理每条事件，                  （批量积累数据后处理，
 批次更小，overhead 更大）             效率更高，延迟更大）
```

**这两者没有绝对的对错。** 正确的平衡点完全取决于你的 SLA（服务级别协议）和业务场景：

| 场景 | 可接受延迟 | 优化方向 |
|---|---|---|
| 欺诈检测 | < 500ms | 延迟优先 |
| 实时看板 | < 5 分钟 | 延迟优先 |
| 每日报表 | < 2 小时 | 吞吐量 / 成本 |
| 历史分析 | < 24 小时 | 吞吐量 / 成本 |
| ML 特征库刷新 | 视场景而定 | 两者兼顾 |

**在不增加成本的前提下，两者无法同时最优化。** 更低的延迟意味着更小的微批次、更频繁的任务触发、更多的调度 overhead——这会推高基础设施成本并降低吞吐效率。

**调优之前，先定义 SLA。** 否则你是在没有终点的情况下奔跑。

## 3. 延迟问题具体出现在哪些场景？

### 3.1 批处理 Pipeline

在经典的 ETL/ELT 批处理架构中，延迟问题通常表现为：

- **任务运行时间爆炸**：原来跑 10 分钟的 Pipeline，随着数据量增长变成了 3 小时。常见原因：全表扫描、缺少过滤条件、低效 Join、没有分区。
- **调度器队列阻塞**：Airflow DAG 计划在凌晨 2 点执行，但上游任务还没完成，导致下游任务空等 slot，什么都不做。
- **数据倾斜（Data Skew）**：Spark 的某个分区持有 80% 的数据。199 个 Task 在 2 分钟内完成，1 个 Task 跑了 45 分钟。整个任务的总耗时由最慢的那个 Task 决定。

### 3.2 流处理 Pipeline

在流处理架构（Kafka、Kinesis、Flink、Spark Structured Streaming）中，延迟通常源于：

- **Consumer Lag（消费者滞后）**：生产者写入的最新 offset 与消费者已处理的 offset 之间的差距。Lag 持续增长，说明消费者跟不上生产者。
- **Trigger 频率设置**：如果 Spark Structured Streaming 的 trigger 每 60 秒触发一次，端到端最小延迟就是 60 秒——无论每条记录的处理速度多快。
- **Watermark 配置**：Watermark 定义了你愿意等多久才关闭时间窗口。较长的 watermark 降低数据丢失风险，但会增加结果输出的延迟。
- **分区瓶颈**：Kafka 的 partition 数量决定消费者最大并行度上限。如果一个 topic 有 4 个 partition，部署 16 个 consumer 实例，其中 12 个实例永远是空闲的。

### 3.3 云数据仓库查询（Snowflake、BigQuery、Redshift）

即使数据已经落到仓库，查询延迟依然是常见问题：

- **冷启动 / 仓库恢复**：Snowflake 的虚拟仓库在空闲一段时间后会自动挂起。冷启动后的第一条查询需要等待仓库恢复，产生额外延迟。
- **全表扫描**：没有合适的 clustering key 或分区裁剪，一条只需要 1% 数据的查询却扫描了整张表。
- **溢写到磁盘（Spill to Disk）**：当一个操作（如大排序或大 Join）超出内存限制时，Snowflake 会将数据溢写到本地磁盘甚至远程存储——比内存操作慢几个数量级。
- **dbt 物化方式不当**：`view` 类型的 dbt 模型在每次被查询时都会重新执行上游所有 SQL。如果 lineage 很复杂，这会非常昂贵。

### 3.4 调度层瓶颈（Airflow）

- **任务串行执行**：本可以并行的任务，因为过于保守的依赖设置而被强制串行。
- **轮询 overhead**：Sensor 检查条件的频率太高（浪费 worker slot）或太低（增加不必要的等待时间）。
- **并行度配置不足**：Airflow worker 数量不够，无法支撑并发任务量。

## 4. 优化框架：先测量，再调优

性能工程中最昂贵的错误是**优化了错误的东西**。

不管任何情况，严格遵守以下顺序：

```
1. 定义 SLA        → 可接受的延迟上限是多少？
2. 测量            → 时间实际消耗在哪里？
3. 定位瓶颈        → 当前最大的单一约束是什么？
4. 提出假设        → 根本原因可能是什么？
5. 只改一个变量    → 不要同时调多个参数
6. 再次测量        → 改善了吗？改善了多少？
7. 重复
```

### 各层级测量工具

| 层级 | 工具 |
|---|---|
| Spark | Spark UI（DAG 视图、Stage 时间线、Task 指标） |
| Kafka | Consumer group lag 指标（`kafka-consumer-groups.sh --describe`） |
| Airflow | Task 运行时长日志、Airflow UI Gantt 视图 |
| Snowflake | Query Profile、`QUERY_HISTORY` 视图、`WAREHOUSE_METERING_HISTORY` |
| dbt | `dbt run` 输出、dbt Cloud 任务运行历史 |
| 通用监控 | Prometheus + Grafana、Datadog、OpenTelemetry |

**只有在定位到瓶颈之后，才开始调参数。**

## 5. 第一层 — 计算层优化

### 5.1 分区与并行度

分区是分布式计算中影响最大的单一调优手段。目标是将数据均匀分配到所有可用 worker，避免任何单个 Task 成为瓶颈。

**在 Spark 中：**

```python
# 分区过少 → 集群利用率低
# 分区过多 → 过量 shuffle overhead 和 Task 调度成本

# 查看当前分区数
df.rdd.getNumPartitions()

# repartition：触发全量 shuffle，用于需要增加分区数时
df = df.repartition(200, "partition_column")

# coalesce：不触发 shuffle，只用于减少分区数（例如写出前合并小文件）
df = df.coalesce(10)
```

常用经验值：每个分区的数据量控制在 100MB 到 200MB 之间。100GB 数据对应约 500～1000 个分区。

**选择分区列的原则**：选择能均匀分布数据的列——时序数据通常用日期列（`event_date`），或使用高基数 ID 列。避免低基数列（例如布尔值字段），否则必然造成倾斜。

### 5.2 处理数据倾斜

数据倾斜是指各分区的数据量分布极度不均。这是导致 Spark 任务莫名其妙慢的最常见原因之一。

**如何发现倾斜**：在 Spark UI 的 Stage 视图中，如果大多数 Task 几秒内完成，但少数几个 Task 运行了几分钟，就是倾斜。

**解决方案：**

```python
# 方案一：加盐（Salting）—— 对倾斜的 key 人为添加随机后缀
from pyspark.sql.functions import concat, col, lit, rand, floor

# 给倾斜的 Join key 加上 0~9 的随机盐值后缀
df_skewed = df_skewed.withColumn(
    "salted_key",
    concat(col("join_key"), lit("_"), floor(rand() * 10).cast("string"))
)

# 将小表复制 10 份，分别对应各个盐值
from pyspark.sql.functions import explode, array

df_small = df_small.withColumn("salt", explode(array([lit(i) for i in range(10)])))
df_small = df_small.withColumn(
    "salted_key",
    concat(col("join_key"), lit("_"), col("salt").cast("string"))
)

# 用 salted_key 做 Join
result = df_skewed.join(df_small, on="salted_key")
```

```python
# 方案二：广播 Join（Broadcast Join）—— 小表 Join 大表时消除 shuffle
from pyspark.sql.functions import broadcast

result = large_df.join(broadcast(small_df), on="join_key")
```

广播 Join 将小表发送到每一个 executor，彻底消除 shuffle。适用条件：小表能放入内存（默认阈值 10MB，可通过 `spark.sql.autoBroadcastJoinThreshold` 配置）。

### 5.3 谓词下推（Predicate Pushdown）

尽可能将过滤逻辑前移到数据源层面，让更少的数据进入计算引擎。

```python
# 反面示例：先全量读取，再在 Spark 中过滤
df = spark.read.parquet("s3://bucket/events/")
df_filtered = df.filter(col("event_date") == "2024-01-15")

# 正确做法：直接读取对应分区路径
df = spark.read.parquet("s3://bucket/events/event_date=2024-01-15/")

# 或者利用 Spark 的原生下推（对分区过的 Parquet/Delta 自动生效）
df = spark.read.parquet("s3://bucket/events/").filter(col("event_date") == "2024-01-15")
```

在列式存储格式（Parquet、ORC）下，谓词下推还能跳过不匹配过滤条件的 row group，直接不读那部分字节。

### 5.4 缓存与持久化

如果同一个 DataFrame 在 Job 中被使用多次，缓存它以避免重复计算。

```python
from pyspark import StorageLevel

# 缓存到内存（默认）
df.cache()

# 或者明确指定持久化级别
df.persist(StorageLevel.MEMORY_AND_DISK)

# 用完之后必须释放——Spark 不会自动回收缓存数据
df.unpersist()
```

**不要盲目缓存。** 缓存一个只使用一次的大型 DataFrame 会浪费内存，甚至触发溢写，反而变慢。满足以下两个条件时才值得缓存：（a）转换计算代价高；（b）结果在下游被使用至少两次。

## 6. 第二层 — 存储与 I/O 优化

### 6.1 文件格式选择

文件格式的选择对读取性能有决定性影响。

| 格式 | 存储类型 | 压缩 | Schema 演进 | 适用场景 |
|---|---|---|---|---|
| CSV / JSON | 行式 | 可选 | 有限 | 仅用于原始数据摄入 |
| Avro | 行式 | 支持 | 强 | Kafka 序列化、写密集型场景 |
| Parquet | 列式 | 支持 | 良好 | 分析查询、读密集型场景 |
| ORC | 列式 | 支持 | 良好 | Hive/Hadoop 生态 |
| Delta Lake | 列式 + ACID | 支持 | 强 | Lakehouse、Upsert、流处理 |

**分析型工作负载，始终使用 Parquet 或 Delta Lake。** 一条只读取 100 列中 3 列的查询，扫描的字节量仅为行式格式的约 3%。

### 6.2 小文件问题（Small File Problem）

流处理或微批任务频繁写入时，会产生大量微小文件。每次文件的打开/关闭操作都有 overhead，读取 10,000 个 1MB 的文件远比读取 10 个 1GB 的文件慢——即使总数据量完全相同。

**症状**：S3 LIST 操作很慢；Spark 读取数据湖时 Task 数量极多，但每个 Task 持续时间极短。

**解决方案：**

```python
# 1. 写出前先 coalesce（减少输出文件数量）
df.coalesce(10).write.mode("append").parquet("s3://bucket/output/")

# 2. 使用 Delta Lake 的 OPTIMIZE 命令做文件合并（定期作为维护任务执行）
spark.sql("OPTIMIZE delta.`s3://bucket/delta-table/`")

# 3. 对于 Spark Structured Streaming，使用 trigger.availableNow
#    将多个微批的输出合并成更少的文件
```

### 6.3 存储层分区

按照查询中最常用的过滤列对数据湖进行分区，目录结构约定如下：

```
s3://bucket/events/
    year=2024/
        month=01/
            day=15/
                part-00000.parquet
                part-00001.parquet
```

当查询过滤条件为 `year=2024 AND month=01` 时，存储层直接跳过所有其他目录——不需要任何计算。

**分区列的选择原则**：基数过高的列（如 `user_id`）会产生太多微小分区；基数过低的列（如 `is_active`）会产生太少的大分区。对时序事件数据，以日期（`event_date` 或 `year/month/day`）做分区是最安全的默认选择。

## 7. 第三层 — 网络与基础设施优化

### 7.1 数据本地性：计算与存储同区域部署

网络 I/O 比内存或本地磁盘慢几个数量级。每一个字节跨网络传输（尤其是跨云区域）都会带来延迟。

**原则：将计算和存储部署在同一个云区域（Region）。**

| 场景 | 延迟影响 |
|---|---|
| EMR（us-east-1）读取 S3（us-east-1） | 极低——同区域 |
| EMR（us-east-1）读取 S3（us-west-2） | 高——跨区域出口流量 |
| Snowflake（us-west-2）查询 S3（us-east-1）上的 External Stage | 高——完全可以避免 |

如果你的 Snowflake 账户在 `us-west-2`（AWS Oregon），S3 bucket 也必须在 `us-west-2`。这不是小细节——跨区域传输不仅带来持续、不可规避的延迟，还会产生按 GB 计费的流量费用。

### 7.2 压缩

压缩以 CPU 时间为代价，减少网络传输的字节量。对数据工程场景，这个权衡几乎总是值得的。

| 编解码器 | 压缩比 | 速度 | 适用场景 |
|---|---|---|---|
| **Snappy** | 中 | 极快 | Spark shuffle 中间数据 |
| **LZ4** | 中 | 极快 | Kafka 消息、低延迟流处理 |
| **Zstd** | 高 | 快 | Parquet 文件、归档存储 |
| **Gzip** | 高 | 慢 | 低频访问的归档文件 |

```python
# 设置 Spark shuffle 压缩算法
spark.conf.set("spark.io.compression.codec", "snappy")

# 写出 Parquet 时指定压缩格式
df.write \
    .option("compression", "zstd") \
    .parquet("s3://bucket/output/")
```

### 7.3 连接池（Connection Pooling）

建立一个新的数据库连接代价很高——涉及认证、会话建立以及两端的资源分配。在频繁发起短查询的场景（例如 Python 脚本逐条处理记录），连接 overhead 会主导总运行时间。

使用连接池来复用已有连接：

```python
# SQLAlchemy 连接池示例
from sqlalchemy import create_engine

engine = create_engine(
    "snowflake://...",
    pool_size=5,        # 维持 5 个持久连接
    max_overflow=10,    # 允许额外创建最多 10 个临时连接
    pool_pre_ping=True  # 使用前验证连接是否存活
)
```

## 8. 第四层 — 流处理专项优化（Kafka + Spark）

本节直接对应 **Kafka → Spark Structured Streaming → Snowflake** 架构。

### 8.1 Kafka Partition 数量与消费者并行度

在 Kafka 中，Topic 的 partition 数量是消费者并行度的**硬性上限**，消费者实例数不能超过 partition 数。

```
Topic: events  |  Partition 数: 4
Consumer Group: spark-consumer  |  实例数: 8

→ 只有 4 个实例在真正消费，另外 4 个实例永远空闲
→ 要提升并行度，必须先增加 partition 数
```

**设计建议：**
- Partition 数量应按照目标吞吐量设计，而不是当前数据量。Partition 一旦创建，事后很难缩减。
- 常用估算公式：`目标吞吐量(MB/s) ÷ 单 Partition 吞吐量(MB/s)`
- 本地开发环境：从 4～8 个 partition 开始，上生产前再按需扩展。

### 8.2 Spark Structured Streaming Trigger 配置

Trigger 决定了 Spark 处理微批的频率，是控制端到端流处理延迟最直接的手段。

```python
from pyspark.sql import SparkSession

spark = SparkSession.builder.getOrCreate()

df = spark.readStream \
    .format("kafka") \
    .option("kafka.bootstrap.servers", "localhost:9092") \
    .option("subscribe", "events") \
    .option("maxOffsetsPerTrigger", 50000) \   # 每个微批最多处理 50000 条记录
    .load()

# 方案一：固定间隔触发
query = df.writeStream \
    .trigger(processingTime="30 seconds") \  # 每 30 秒触发一次
    .outputMode("append") \
    .format("parquet") \
    .option("path", "s3://bucket/output/") \
    .start()

# 方案二：连续处理（实验性，极低延迟）
query = df.writeStream \
    .trigger(continuous="1 second") \  # 每 1 秒做一次 checkpoint
    .outputMode("append") \
    .format("console") \
    .start()

# 方案三：availableNow（类批处理调度，处理完当前所有可用数据后停止）
query = df.writeStream \
    .trigger(availableNow=True) \  # Spark 3.3+
    .outputMode("append") \
    .format("parquet") \
    .start()
```

**Trigger 权衡对比：**

| Trigger 间隔 | 延迟 | 吞吐效率 | 基础设施成本 |
|---|---|---|---|
| 1 秒 | 极低 | 低 | 高 |
| 30 秒 | 低 | 中 | 中 |
| 5 分钟 | 中 | 高 | 低 |
| `availableNow` | 高（按调度） | 极高 | 极低 |

### 8.3 Watermark 与迟到数据

Watermark 告诉 Spark 在关闭时间窗口并输出结果之前，最多等待多久的迟到数据。这直接控制**数据完整性 vs. 结果延迟**的权衡。

```python
from pyspark.sql.functions import window, col

windowed_counts = df \
    .withWatermark("event_timestamp", "10 minutes") \  # 最多等待 10 分钟的迟到数据
    .groupBy(
        window(col("event_timestamp"), "5 minutes"),   # 5 分钟滚动窗口
        col("user_id")
    ) \
    .count()
```

- **短 Watermark**（如 `"2 minutes"`）：结果输出更快，但可能丢失迟到事件，导致数据不完整。
- **长 Watermark**（如 `"1 hour"`）：数据更完整，但结果输出相应延迟一小时。

**Watermark 应该根据你观测到的实际数据到达模式来设置**，而不是凭感觉。检查事件投递 Pipeline 的 P99 延迟，了解事件实际上最晚会晚到多久。

### 8.4 Kafka Consumer 调优参数

以下是影响吞吐量和延迟的关键 Consumer 配置参数：

```python
# 这些参数通过 Spark Kafka Source 的 .option() 方式传入
options = {
    "kafka.fetch.min.bytes": "1048576",       # 1MB — 每次 fetch 至少等到有 1MB 数据
    "kafka.fetch.max.wait.ms": "500",          # fetch.min.bytes 未满足时最多等 500ms
    "kafka.max.poll.records": "1000",          # 每次 poll 最多取 1000 条记录
    "kafka.session.timeout.ms": "30000",       # Consumer 会话超时时间
    "startingOffsets": "latest",               # 首次启动从最新位置开始消费
}
```

- `fetch.min.bytes` / `fetch.max.wait.ms`：控制 fetch 请求的批量行为。`fetch.min.bytes` 越大，吞吐量越高，但延迟也越大。
- `max.poll.records`：每次 poll 获取的记录数。处理速度快时可以调高；处理较慢时调低，避免触发 Consumer Group Rebalance。

## 9. 调度层优化（Airflow）

### 9.1 最大化 DAG 内的并行度

最常见的 Airflow 延迟问题：给本可并行的任务设置了不必要的串行依赖。

```python
# 错误示范：任务之间其实互相独立，却被强制串行执行
extract_users >> extract_orders >> extract_products >> transform

# 正确做法：独立任务并行执行，汇聚后再执行 transform
from airflow.utils.task_group import TaskGroup

with TaskGroup("extract") as extract_group:
    extract_users = PythonOperator(...)
    extract_orders = PythonOperator(...)
    extract_products = PythonOperator(...)

extract_group >> transform  # 三个任务并行，transform 等待全部完成后执行
```

### 9.2 使用 Deferrable Operator 处理外部等待

普通 Sensor 在轮询期间会占用一个 Airflow worker slot，即使什么都不做。Deferrable Operator（Airflow 2.2 引入）在等待期间释放 worker slot，由 triggerer 进程负责唤醒。

```python
# 普通 Sensor：每 30 秒轮询一次 S3，期间持续占用 worker slot
from airflow.providers.amazon.aws.sensors.s3 import S3KeySensor

wait_for_file = S3KeySensor(
    task_id="wait_for_file",
    bucket_name="my-bucket",
    bucket_key="data/{{ ds }}/file.parquet",
    poke_interval=30,
    mode="poke"  # 占用 worker slot
)

# Deferrable 模式：等待期间释放 worker slot，由 triggerer 进程负责恢复
wait_for_file = S3KeySensor(
    task_id="wait_for_file",
    bucket_name="my-bucket",
    bucket_key="data/{{ ds }}/file.parquet",
    deferrable=True  # 需要 Airflow triggerer 进程运行
)
```

### 9.3 Pool 与优先级配置

使用 Airflow Pool 限制对外部资源的并发访问（如 Snowflake 连接数、API 速率限制），并设置任务优先级确保关键路径任务优先被调度。

```python
transform_task = SnowflakeOperator(
    task_id="transform_large_table",
    sql="...",
    pool="snowflake_pool",      # 来自此 Pool 的 Snowflake 任务最多同时 5 个
    priority_weight=10,          # 优先级高于默认任务
)
```

## 10. 云数仓中的延迟优化（Snowflake）

### 10.1 Clustering Key（聚簇键）

Snowflake 将数据组织成 micro-partition（每个最大 16MB）。没有聚簇键时，一条过滤 `event_date` 的查询必须扫描所有 micro-partition。定义了聚簇键后，Snowflake 在扫描前会先裁剪掉不相关的 micro-partition。

```sql
-- 在常用过滤列上定义聚簇键
ALTER TABLE fact_events CLUSTER BY (event_date);

-- 检查聚簇健康状况
SELECT SYSTEM$CLUSTERING_INFORMATION('fact_events', '(event_date)');
```

适合使用聚簇键的条件：
- 表数据量超过约 1TB
- 查询始终过滤相同的 1～2 列
- 这些列的基数**适中**（日期、地区、产品类别——不适合用 user_id）

### 10.2 仓库规格与自动挂起

Snowflake 虚拟仓库规格过小时，内存不足的操作会溢写到本地磁盘乃至远程存储。溢写到远程存储可能比内存操作慢 100 倍。

```sql
-- 在 Query Profile 中检查 "Bytes spilled to local/remote storage"
-- 如果发现大量远程溢写，需要提升仓库规格

ALTER WAREHOUSE my_warehouse SET WAREHOUSE_SIZE = 'LARGE';

-- 通过自动挂起控制空闲成本
ALTER WAREHOUSE my_warehouse SET
    AUTO_SUSPEND = 60          -- 空闲 60 秒后自动挂起
    AUTO_RESUME = TRUE;        -- 有查询时自动恢复
```

### 10.3 dbt 物化策略

在 dbt 中，物化方式决定了下游消费者是在定义时还是在查询时支付计算成本。

```yaml
# dbt_project.yml — 按模型层级设置物化方式

models:
  my_project:
    staging:
      +materialized: view          # 轻量，始终最新，但每次查询都重新执行上游 SQL
    intermediate:
      +materialized: ephemeral     # 无物理对象，内联进下游 SQL
    marts:
      +materialized: table         # 预计算，查询快，dbt run 时刷新
      +cluster_by: ["event_date"]  # Snowflake 专用：对输出表做聚簇
```

对于查询频繁的大型 Fact 表，`table` 物化方式几乎总是比 `view` 更低的查询延迟——代价是更长的 `dbt run` 时间。

对于频繁更新的增量模型：

```sql
-- models/marts/fact_events.sql
{{
    config(
        materialized='incremental',
        unique_key='event_id',
        cluster_by=['event_date']
    )
}}

SELECT * FROM {{ ref('stg_events') }}

{% if is_incremental() %}
WHERE event_date >= (SELECT MAX(event_date) FROM {{ this }})
{% endif %}
```

## 11. 真实场景还原与排查过程

### 场景 A：批处理任务从 20 分钟涨到 4 小时

**症状**：一个夜间 Spark 任务处理 10GB 数据时只需 20 分钟，数据量增长到 80GB 后变成了 4 小时。

**排查过程：**
1. 查看 Spark UI → Stage 时间线显示某个 Stage 耗时 3 小时 50 分，其他 Stage 都很快
2. 该 Stage 中，199 个 Task 在 1 分钟内完成，1 个 Task 跑了 3 小时 40 分
3. **根本原因：数据倾斜。** 某个 Join key 的特定值持有 60% 的全部数据

**修复措施：**
- 对倾斜的 Join key 添加盐值（Salting）
- 分区数从 200 增加到 600
- 结果：任务运行时间降至 35 分钟

**经验教训**：大多数 Task 很快、少数 Task 很慢 → 是数据倾斜，不要增加集群规模，要修分布。

---

### 场景 B：Kafka Consumer Lag 持续增长

**症状**：Spark Structured Streaming 任务部署后，Consumer Lag 从 0 开始，每分钟增长约 5000 条消息。

**排查过程：**
1. Consumer Lag 指标持续增长 → 消费者跟不上生产者
2. Spark UI 显示每个微批处理耗时 45 秒，而 Trigger 间隔设置的是 30 秒
3. 批处理时间 > Trigger 间隔 → 永久性落后，无法追上
4. Executor 指标：CPU 100%，无溢写，Shuffle 很少

**根本原因**：新增事件类型后数据量增长了 3 倍，现有处理逻辑对于当前 Executor 资源来说过于 CPU 密集。

**修复措施：**
- Spark 集群新增 4 个 Executor（水平扩展）
- `maxOffsetsPerTrigger` 从 10,000 提升至 30,000，让更大的批次分摊调度 overhead
- 临时将 Trigger 间隔从 30 秒调整到 60 秒，给积压消息追赶时间
- 结果：Consumer Lag 在 2 小时内清零，稳定在 0

**经验教训**：批处理时间持续超过 Trigger 间隔，是需要更多计算资源的明确信号，而不是调参问题。

---

### 场景 C：Snowflake 查询从 5 秒退化到 4 分钟

**症状**：一个服务于 BI 看板的 Mart 查询，在没有任何代码改动的情况下，从约 5 秒退化到约 4 分钟。

**排查过程：**
1. Query Profile 显示 `TableScan` 阶段占据 98% 的耗时
2. `QUERY_HISTORY` 中 `BYTES_SCANNED` 从 2GB 暴增到 180GB
3. 检查 dbt 模型：一次近期 PR 将该 Mart 模型的物化方式从 `table` 改成了 `view`

**根本原因**：将 Mart 模型改为 `view` 后，看板每次查询都会重新执行整个上游 dbt lineage，而不是从预计算好的表中读取结果。

**修复措施：**
- 将物化方式恢复为 `table`
- 新增 `cluster_by: ["report_date"]` 改善 micro-partition 裁剪效果
- 结果：查询时间回到约 6 秒

**经验教训**：dbt 的物化方式选择是性能决策，不仅仅是存储决策。在不理解下游查询模式的情况下，不要随意修改它。

## 12. 总结与决策清单

遇到生产环境中的延迟问题时，使用以下清单：

```
动手之前：
☐ 定义目标 SLA（"可接受"的标准是什么？）
☐ 确认是哪个指标在退化（端到端延迟？吞吐量？查询时间？）
☐ 确认问题真实存在且可复现

测量阶段：
☐ 端到端 Profile Pipeline——时间实际消耗在哪里？
☐ 检查常见嫌疑点：数据倾斜、全表扫描、Consumer Lag、磁盘溢写

计算层：
☐ 分区数是否与数据量和 Worker 数匹配？
☐ 是否存在数据倾斜？（检查 Spark UI Task 耗时的方差）
☐ Join 是否已优化？（适用场景下是否使用了广播 Join？）
☐ 缓存使用是否正确？

存储层：
☐ 是否使用了列式存储格式（Parquet、ORC、Delta）？
☐ 是否存在小文件问题？
☐ 存储分区是否与查询过滤模式对齐？

网络层：
☐ 计算和存储是否在同一个云区域？
☐ 网络传输和存储文件是否启用了压缩？

流处理（如适用）：
☐ Kafka Partition 数量是否 ≥ Consumer 实例数？
☐ Trigger 间隔是否符合 SLA 要求？
☐ Watermark 时长是否基于实际迟到数据到达模式来设置？
☐ Consumer Lag 是稳定的还是持续增长的？

数据仓库：
☐ 高频过滤列是否定义了聚簇键？
☐ 仓库规格是否足够，避免磁盘溢写？
☐ dbt 物化方式是否与查询访问模式匹配？

调度层：
☐ 独立任务是否在并行执行？
☐ Sensor 是否使用了 Deferrable 模式？
☐ Pool 大小和任务优先级是否已正确配置？
```

## 结语

延迟优化本质上是一门**系统性测量的学科，而不是凭直觉猜测的艺术**。排查性能问题最快的工程师，不是那些记住了最多配置参数的人，而是那些能快速判断问题所在层级、提出可验证的假设并用数据来验证它的人。

每一次优化决策都应该回答这三个问题：

1. **时间实际消耗在哪里？**（测量）
2. **当前最大的单一约束是什么？**（定位瓶颈）
3. **解决它的代价是什么？**（延迟、吞吐量、基础设施成本、工程投入）

其他所有内容，都从这三个问题推导而来。
